# 自动回复架构说明

本文档记录当前微信自动回复系统的主要架构、配置和处理流程。

## 1. 整体分工

当前系统分为三块：

1. `MonitorService`
   负责监听微信好友消息，并把收到/发送的消息写入数据库。

2. `OpenAIAutoReplyService`
   负责定时检查未读消息，调用 OpenAI-compatible 接口生成回复结果。

3. `WeChatService`
   负责实际发送微信消息，并在需要时保存发送记录。

Hermes Agent 不再用于自动回复。自动回复改为轻量 OpenAI-compatible API 调用。Hermes 后续只适合用于主动发消息、复杂 Agent 行为等场景。

## 2. 配置文件

配置文件位置：

```txt
WebApi/data_config.json
```

当前结构：

```json
{
  "OpenAI": {
    "url": "https://api-inference.modelscope.cn/v1",
    "key": "xxx",
    "model_name": "moonshotai/Kimi-K2.5"
  },
  "FallBack": {
    "url": "",
    "key": "",
    "model_name": ""
  },
  "friends": [
    {
      "monitor_friend_name": "佳霖",
      "is_monitoring": true,
      "Avatar": "A角色"
    }
  ]
}
```

字段说明：

- `OpenAI`: 主模型供应商配置。
- `FallBack`: 备用模型供应商配置，主供应商失败后自动尝试。
- `friends`: 好友监听配置。
- `monitor_friend_name`: 微信好友名称。
- `is_monitoring`: 是否监听该好友。
- `Avatar`: 使用哪个角色 system prompt，对应 `WebApi/avatars/{Avatar}.md`。

## 3. 角色与背景

角色文件：

```txt
WebApi/avatars/A角色.md
WebApi/avatars/B角色.md
```

角色文件就是 system prompt，负责定义回复规则、输出 JSON 格式、事实边界等。

好友背景文件：

```txt
WebApi/avatars/{好友名}.md
```

角色文件中可以使用：

```txt
{{avatar_background}}
```

运行时会替换成对应好友的背景信息。

## 4. 自动回复流程

基本流程：

```txt
监听微信消息
-> 写入数据库
-> 自动回复线程轮询未读消息
-> 读取好友配置和角色 prompt
-> 构造 OpenAI messages
-> 调用 OpenAI-compatible /chat/completions
-> 解析 JSON
-> 自动回复 / 不回复 / 发给文件传输助手
```

发送给模型的结构大致是：

```python
[
  {"role": "system", "content": "角色 prompt + 好友背景"},
  {"role": "user", "content": "好友历史消息"},
  {"role": "assistant", "content": "我方历史回复"},
  {"role": "user", "content": "好友最新未读消息"}
]
```

聊天记录不再拼进 prompt 文本，而是作为标准 `messages` 上下文传给模型。

## 5. AI 返回格式

模型必须返回 JSON：

```json
{
  "should_reply": true,
  "reply_text": "这里填写自动回复内容",
  "need_human": false,
  "suggested_reply": "",
  "reason": "这里填写简短原因",
  "confidence": 0.88
}
```

处理规则：

- `should_reply=true`: 自动发送 `reply_text` 给好友。
- `should_reply=false`: 不回复，直接标记已读。
- `need_human=true`: 不自动回复好友，把建议回复发给“文件传输助手”。

发给文件传输助手的格式：

```txt
张三消息: 你现在在干嘛?   建议回复:
刚看到消息，你找我有事吗？
```

文件传输助手消息不会写入数据库。

## 6. 数据库规则

数据库文件：

```txt
WebApi/wechat.db
```

主要表：

```txt
chat_messages
```

当前字段包括：

- `friend_name`
- `received_messages`
- `sent_messages`
- `is_read`
- `received_time`
- `sent_time`
- `remark`

消息合并规则：

- 好友连续发消息，会用 ` ; ` 合并。
- 如果最新记录还没有我方回复，即使已读，后续好友消息也会继续合并到这一行。
- 我方发送消息会优先合并到该好友最新记录。

已加索引：

```sql
idx_chat_friend_id
idx_chat_friend_unread
idx_chat_unread
```

这些索引用于加速按好友查历史、按好友查未读、查全部未读。

## 7. 当前注意事项

- 当前数据库结构能用，但不是长期最理想结构。
- 如果后续功能复杂起来，建议拆成 `messages` 和 `conversations` 两张表。
- 当前 OpenAI 调用会打印完整请求和响应，方便调试；稳定后可以注释掉相关 `print`。
- `FallBack` 为空时不会启用备用供应商。
